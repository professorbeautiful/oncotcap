Inheritance of parameter values

A key idea in OncoTCap is that models should be described using words, that is, in the textual model-driving statement contained in a Statement Template.
It should be easy to judge whether a model-driving statement
   1 accurately states the intention (looking "upstream" at the Edict), and
   2 is implemented faithfully (looking "downstream" at the supporting instruction-providers).

To make this efficient, statements must be re-usable.
This is accomplished by two techniques:
   1 The value of each parameter, assigned in the Statement Bundle by the author or the end-user, is passed "downstream" to the supporting instruction-providers. The Instruction-providers may include the name of the parameter surrounded by back-quotes. When it comes time to run the model, this parameter in back-quotes is replaced by looking to the "upstream" Statement Bundle. Each model computational engine will have a translator to read the contents of the parameter value and write out the engine-specific code. In the simulator which is native to (included in) OncoTCap, that translator is the JavaCodeProvider class.

   2 Statement Templates are implemented ultimately by Code bundles, but they can also be implemented by Statement Bundles that in turn are based upon other Statement Templates. This allows the packaging of multiple simple ideas into more complex ones.
